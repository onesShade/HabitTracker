## Введение

Данный документ описывает план тестирования веб-приложения **"Habit loop"**. Цель тестирования — убедиться, что все функциональные и нефункциональные требования, описанные в документе SRS, реализованы корректно. Особое внимание уделяется правильности работы механик учёта привычек, визуализации данных в календаре и безопасности пользовательских данных.

<a name="items"/>

## Объект тестирования

Объектами тестирования являются ключевые функциональные модули приложения:

*   Регистрация и аутентификация пользователя;
*   Управление привычками (создание, удаление, просмотр списка);
*   Механизм отслеживания выполнения привычек (ФИП — Факт исполнения привычки);
*   Визуализация прогресса в календаре привычек;
*   Управление годовыми целями;
*   Корректность работы пользовательского интерфейса на всех ключевых экранах.

<a name="quality"/>

## Атрибуты качества

1.  **Функциональность**
    *   Все заявленные функции (добавление, удаление, отметка) работают без сбоев.
    *   Календарь точно отображает отмеченные дни для каждой привычки.
    *   Данные пользователя корректно сохраняются и загружаются при входе с разных устройств.

2.  **Удобство использования (Usability)**
    *   Интерфейс интуитивно понятен и соответствует макетам, представленным в SRS.
    *   Процесс добавления привычки и отметки её выполнения является простым и быстрым.
    *   Навигация между экранами (список привычек, календарь, профиль) логична.

3.  **Надежность**
    *   Приложение стабильно работает при некорректном вводе данных (например, пустые поля в форме).
    *   Данные не теряются при обрыве интернет-соединения или закрытии вкладки браузера.
    *   Система корректно обрабатывает пограничные случаи (например, отметка привычки в полночь).

4.  **Безопасность**
    *   Пароли пользователей хранятся в зашифрованном виде.
    *   Данные одного пользователя полностью изолированы и недоступны другому.
    *   Реализована защита от базовых веб-уязвимостей (например, XSS в полях для заметок).

5.  **Производительность**
    *   Время отклика интерфейса на действия пользователя не превышает 1 секунды, как указано в SRS.
    *   Загрузка списка привычек и календаря происходит быстро, даже при большом количестве записей.

<a name="risk"/>

## Риски

Основные риски проекта:

*   **Потеря данных** при сбоях в работе базы данных H2 или некорректном завершении сессии.
*   **Ошибки в отображении данных** в календаре, особенно при большом количестве привычек.
*   **Проблемы с безопасностью**, если данные пользователей не шифруются должным образом.
*   **Низкая производительность** при увеличении числа пользователей и объёма данных.

<a name="features"/>

## Аспекты тестирования

Тестирование охватывает все основные сценарии использования системы, описанные в Use Case.

### Регистрация и аутентификация

*   Проверка регистрации нового пользователя с корректными и некорректными данными (занятый email, короткий пароль).
*   Проверка входа в систему с правильными и неверными учётными данными.
*   Тестирование функции выхода из системы и завершения сессии.

### Управление привычками

*   Создание новой привычки (положительной и отрицательной).
*   Просмотр списка всех добавленных привычек.
*   Удаление существующей привычки и проверка, что все связанные данные (отметки) также удаляются.

### Отслеживание прогресса

*   Отметка выполнения привычки на текущую дату.
*   Добавление заметки к выполненной привычке.

### Календарь и цели

*   Проверка корректности отображения иконок выполненных привычек в календаре.
*   Тестирование фильтрации календаря по конкретной привычке.
*   Проверка отображения всплывающего окна с заметкой.
*   Создание, просмотр и удаление годовой цели в профиле пользователя.

<a name="nonfunctional"/>

## Нефункциональные требования

*   Интерфейс должен корректно отображаться в последних версиях популярных браузеров (Chrome, Firefox).
*   Время отклика сервера < 1 секунды.
*   Uptime сервера > 99%.
*   Приложение требует постоянного подключения к интернету для синхронизации данных.

<a name="approach"/>

## Подходы к тестированию

Применяется **комбинированный подход**:

*   **Ручное тестирование** — проверка пользовательского интерфейса, удобства использования, визуального соответствия макетам.
*   **Автоматизированное тестирование** — юнит-тесты для ключевых методов сервисов (сохранение данных), как указано в SRS.
*   **Функциональное тестирование** — проверка всех сценариев использования, описанных в Use Case.
*   **Регрессионное тестирование** — повторное выполнение ключевых тестов после внесения изменений в код.
*   **Нагрузочное тестирование** (упрощённое) для проверки производительности под нагрузкой.
